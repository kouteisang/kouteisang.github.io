react中的fiber

虚拟dom

什么是虚拟dom

react使用虚拟dom，虚拟dom其实就是用JS对象表示的一个dom节点，内部包含了节点的tag，props和children。

为何使用虚拟dom是因为直接操作真是dom十分低效。

如何将JSX转换成虚拟dom

通过babel可以将jsx编译为特定的JS对象。

从babel的编译结果可以看出，虚拟dom中包含了创建dom所需要的各种信息，对于首次渲染，直接按照这些信息创建dom即可。

但是虚拟dom的真正价值在于更新，当一个list中的某项发生了变化，或者删除或增加了若干项，通过对比前后的虚拟dom树，最小化的更新真实dom。这就是react的核心目标。

  1. 虚拟dom是一个object一般对象
  2. 虚拟dom比较”轻量”, 而真实dom比较重（这里所说的轻重指的是虚拟dom中所包含的属性较少，而真实dom包含的属性很多），因为虚拟dom是react内部在用，无需真实dom上那么多属性。
  3. 虚拟dom最终会被react转化为真实dom，呈现在页面上。

react diffing算法

React最大的优势是不是每一次都对页面上的真实dom作出修改，每一个真实dom都对应了虚拟dom的，当生成了新的虚拟dom，react会将新的虚拟dom和旧的虚拟dom进行比较，比较之后没有变化的数据页面上的真实dom是没有任何改变的。只把改变的映射成新的真实dom放到页面上。

react diffing算法对比的最小粒度是标签，如果存在标签的嵌套，diffing算法会对标签进行逐层对比。

经典面试题目：在遍历的时候react/vue中的key有什么作用？

简单的说：key是虚拟dom对象的标识，在更新显示时key起着极其重要的作用。

详细的说：当状态中的数据发生变化时，react会根据【新数据】生成【新的虚拟DOM】，随后React进行【新虚拟DOM】与【旧虚拟DOM】的diff比较。

  1. 若旧虚拟DOM中找到了与新虚拟DOM相同的key：

  • 若虚拟DOM中的内容没变，直接使用之前的真实DOM

  • 若虚拟DOM中的内容变了，则生成新的真实DOM，随后替换掉页面中之前的真实DOM

  1. 旧虚拟DOM中未找到与新虚拟DOM相同的key
  • 根据数据创建新的真实DOM，随后渲染到页面

用index作为key可能会引发的问题

  1. 若对数据进行：逆序添加、删除等破坏顺序操作会产生没有必要的真实DOM更新 ==》 界面效果没问题，但是效率低下。

  2. 如果结构中还包括输入类的DOM，会产生错误DOM更新 ==》界面有问题。

  3. 注意！如果不存在对数据的逆序添加、逆序删除等破坏顺序的操作，仅仅用于旋渲染列表用于展示，使用index作为key是没有问题的。

  4. 开发时最好使用每条数据的唯一标识作为key，比如id。

React Fiber

Fiber作用

支持虚拟DOM的渐进式渲染，Fiber 将原有的 Stack Reconciler 替换为 Fiber Reconciler，提高了复杂应用的可响应性和性能。

为什么会出现Fiber

React15中stack reconciler是通过递归更新子组件。由于递归执行，所以一旦开始，中途就无法中断。当层级很深的时候，递归更新时间超过了16ms，用户交互就会卡顿。（一帧更新60次，平均下来一帧16ms）

我们都知道react 是基于虚拟dom 来分析页面中的dom 元素的变化的，组件需要更新的时候，react 需要同步计算虚拟dom 的变化–>对比新旧虚拟dom–>更新真实dom,因为是同步的，所以当一个组件深层嵌套的时候，庞大的同步计算量会让浏览器无暇去处理别的页面交互，比如说相应用户的input 输入等等你 原来写好的交互事件，用户会觉得页面明显卡顿。

简单理解就是把一个耗时长的任务分解为一个个的工作单元（每个工作单元运行时间很短，不过总时间依然很长）。在执行工作单元之前，由浏览器判断是否有空余时间执行，有时间就执行工作单元，执行完成后，继续判断是否还有空闲时间。没有时间就终止执行让浏览器执行其他任务（如 GUI 线程等）。等到下一帧执行时判断是否有空余时间，有时间就从终止的地方继续执行工作单元,一直重复到任务结束。

要知道每一帧的空闲时间，就需要使用 requestIdleCallback Api。传入回调函数，回调函数接收一个参数（剩余时间），如果有剩余时间，那么就执行工作单元，如果时间不足了，则继续requestIdleCallback，等到下一帧继续判断。

在这种架构下会首先执行优先级高的任务例如页面动画等，防止页面卡顿，给用户好的体验。这些通过requestAnimationFrame(animation)进行执行，执行完毕，通过requestIdleCallBack()查看是否还有剩余的时间，有的话再执行其他任务。因为Fiber是链表结构，所以Fiber可以中断，在有时间的时候再次执行。

什么是Fiber？

  1. Fiber其实指的是一种数据结构，它可以用一个纯JS对象来表示。
  1. Fiber还是一个执行单元，每次执行完一个执行单元，React就会检查现在还剩多少时间，如果没有时间就将控制权让出去。

Fiber关键特性

  1. 增量渲染：将渲染任务拆分到每一帧里面去执行。
  2. 暂停终止复用渲染任务
  3. 不同更新的优先级：在更新时赋予不同任务不同的优先级，高优先级的比如键盘的输入事件，我们希望立马得到反馈
  4. 并发方面新的基础能力：React可以暂停高消耗非紧急的组件渲染，聚焦在更紧迫的任务中， 例如UI渲染，使应用保持可应用高响应，避免白屏、卡顿等现象。

Fiber执行阶段

Fiber每次渲染有两个阶段：Reconciliation(协调render阶段)和commit(提交阶段)

Reconciliation：这个阶段可以认为是diff阶段，这个阶段可以被终止。在reconciliation阶段会又一个current tree，在进行组件更新后，比如setState()后会构建一个workInProgress tree，这个workInProgress tree会复用current tree中未发生变化的fiber（一个fiber就代表了一个元素）。这个阶段会找出所有变更的节点，例如节点新增、删除、属性变更等等，这些变更react称之为side-effects。

提交阶段：将上一阶段计算出来的需要处理的副作用(effects)一次性执行了。这个阶段必须同步执行，不能被打断。workInprogree tree会变成current tree
